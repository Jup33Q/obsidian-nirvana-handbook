## 🧱 Stage 1: Obsidian Raw Material Preprocessing

### 1.1 Material Requirements

Your Obsidian notes must meet the following conditions before entering the nirvana pipeline:

| Check Item | Requirement | Purpose |
|------------|-------------|---------|
| YAML Frontmatter | Must include `title`, `tags`, `date` | For downstream Agent theme detection and metadata extraction |
| Image References | Use WikiLink `![[image.png]]` or standard Markdown `![](path)` | Ensures Marp can correctly parse and embed images |
| Code Blocks | Wrapped in ``` with language annotation | Preserves syntax highlighting after export |
| Heading Hierarchy | Clear H1-H3 sections | Auto-maps to slide pagination logic |

### 1.2 Recommended Folder Structure

```
Obsidian Vault/
├── 01-raw-notes/           # Raw notes, write freely
│   ├── project-review-2024Q4.md
│   └── tech-share-kafka.md
├── 02-nirvana-middleware/  # Marp-ready drafts generated by Agent
│   └── (auto-generated by Agent)
├── 03-slide-outputs/       # HTML/PDF exported by Marp CLI
│   └── (auto-generated by Marp CLI)
└── 04-web-live/            # Phoenix LiveView project
    └── (taken over by Elixir code)
```

### 1.3 Related Skill References

The Obsidian toolchain can directly invoke the following installed Skills:

| Capability | Recommended Skill | Purpose |
|------------|-------------------|---------|
| Quick Template Export | `obsidian-templater` | One-click injection of notes into middleware folder |

> 💡 **One-liner**: Fatten up your notes in Obsidian first, then feed them into the nirvana furnace.

## 🤖 Stage 2: Agent Marp Adaptation (Alchemy Core)

### 2.1 Prompt Template for the Agent

```text
You are a Marp slide architect. Convert the following Obsidian note into Marp-syntax Markdown.

## Input
- Raw Obsidian Markdown (see below)
- Theme preference: {{theme}} (e.g., default, gaia, uncover)
- Pagination strategy: {{strategy}} (e.g., per-H2, auto, manual)

## Output Requirements
1. Insert Marp configuration in Frontmatter:
   ---
   marp: true
   theme: {{theme}}
   paginate: true
   ---

2. Auto-insert pagination divider `---` before H2 (per strategy)
3. Convert WikiLink images to relative paths `![](../assets/xxx.png)`
4. Auto-fold or split code blocks exceeding 20 lines
5. Generate speaker notes `<!-- note content -->` at the bottom of each slide

## Raw Content
{{obsidian_raw_content}}
```

### 2.2 Batch Processing Strategy

Feed `01-raw-notes/` in bulk to the Agent, outputting to `02-nirvana-middleware/marp-ready/`. Use any automation method (GitHub Actions, local scripts, or process page-by-page in conversation).

Core principle: The Agent is responsible **only for syntax conversion** (Obsidian Markdown → Marp Markdown), not for stylistic design.

## 🎨 Stage 3: Marp CLI HTML Export

### 3.1 Installation & Export

```bash
# Install
npm install -g @marp-team/marp-cli

# Single-file export
marp input.md --html --allow-local-files --bundle -o output.html

# Batch export
marp --config-file marp.config.js *.md
```

Key parameters:
- `--html`: Allow HTML tags (required for downstream LiveView embedding)
- `--allow-local-files`: Resolve local images
- `--bundle`: Inline all resources into a single HTML file for easy transfer

### 3.2 Export Artifacts

```
03-slide-outputs/
├── project-review-2024Q4.html      # Single file (embedded CSS/images)
├── tech-share-kafka.html
└── assets/                          # If not bundled, external resources
```

## 🔥 Stage 4: Elixir + Phoenix + LiveView in WSL

> This is the final step of nirvana: turn static HTML slides into a **real-time, controllable, multi-user collaborative** web application.

### 4.1 Environment Setup (WSL2 Ubuntu)

```bash
# Enter WSL
wsl ~

# Elixir / Erlang (asdf version management recommended)
asdf install erlang latest
asdf install elixir latest

# Phoenix
mix local.hex
mix archive.install hex phx_new

# Create new project (with database)
mix phx.new SlideLive
# Or without database
mix phx.new SlideLive --no-ecto --no-mailer
```

### 4.2 Post-Import Initialization in WSL

After copying or `git clone`-ing an existing Phoenix project into WSL, run the following initialization steps:

```bash
# 1. Enter project directory
cd /path/to/slider_live_demo

# 2. Install Elixir dependencies
mix deps.get

# 3. Create and migrate database (if using Ecto)
mix ecto.setup
# Or step-by-step:
# mix ecto.create && mix ecto.migrate && mix run priv/repo/seeds.exs

# 4. Install Node dependencies (assets directory)
cd assets && npm install && cd ..

# 5. Build frontend assets (dev mode: esbuild + tailwatch auto-watch)
# Recommend a manual build before first start
mix assets.build

# 6. Start Phoenix server
mix phx.server
# Or with interactive shell
iex -S mix phx.server
```

#### WSL Network Access Notes

| Access Method | Address | Notes |
|---------------|---------|-------|
| Browser inside WSL | `http://localhost:4000` | Available when lynx/chromium installed in WSL |
| Windows browser | `http://localhost:4000` | WSL2 auto-forwards ports; most common |
| Other LAN devices | `http://<WSL_IP>:4000` | Get IP via `wsl hostname -I` |

> ⚠️ If Windows side cannot access, check Windows Defender Firewall or WSL `autoProxy` settings.

#### Common Development Commands

```bash
# Reset database (clean slate during development)
mix ecto.reset

# Format code (mandatory before commit)
mix format

# Run tests
mix test

# Production build (before deployment)
mix assets.deploy

# Generate new LiveView
mix phx.gen.live Slides Slide slides number:integer title:string
```

### 4.5 Adaptation Rough Steps (Checklist)

Minimum necessary steps when migrating Marp HTML to Phoenix LiveView:

```
□ 1. Initialize project in WSL (see 4.2)
□ 2. Run mix test to ensure baseline passes
□ 3. Create Slide Ecto Schema + Migration + Context
□ 4. Write priv/repo/seeds.exs to pump Marp slide data into DB
□ 5. Create SlideDeckLive (single-page mode, push_navigate pagination)
□ 6. De-inline Marp HTML CSS and migrate into assets/css/app.css
□ 7. Replace SVG foreignObject wrappers with pure HTML <div class="slide">
□ 8. Create BubbleQuilt / TransitionFlash JS Hook (or reference existing Skill)
□ 9. Add keyboard event listeners (← → to paginate, ESC to home)
□ 10. Configure Cookie to remember last_slide, so refresh doesn't lose position
□ 11. Check and remove all external CDN dependencies (twemoji, etc.)
□ 12. Add prefers-reduced-motion media query
□ 13. Run mix test again, ensure all green
□ 14. Verify in Windows browser at localhost:4000
```

## 🛠️ Troubleshooting Quick Reference

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Marp images not showing | Wrong path or missing `--allow-local-files` | Check relative paths, add `--allow-local-files` |
| LiveView not updating | WebSocket blocked by proxy | Check Nginx/firewall ws:// forwarding |
| WSL port inaccessible | WSL network isolation | Use `host.docker.internal` or access localhost from within WSL |
| Slide styles lost | Marp theme not exported correctly | Use `--bundle` to inline CSS, or ensure theme file exists |
| Agent pagination chaotic | Inconsistent H2 hierarchy | Standardize heading levels in Obsidian, or switch to manual pagination |
| HTML file too large | Repeated CSS inlining | See `Marp_to_LiveView_SKILL.md` P1 fix |
| Emoji offline invisible | External CDN dependency | See `Marp_to_LiveView_SKILL.md` Emoji component chapter |
| `mix deps.get` fails | Hex mirror or network issue | `mix hex.config mirror_url https://...` |
| `ecto.setup` errors | SQLite3 not installed | `sudo apt install sqlite3 libsqlite3-dev` |
| Asset build fails | Node version too low | Upgrade Node to 18+, `nvm install 20` |
| Canvas background not showing | Hook not registered or canvas size is 0 | Check `app.js` hooks object + CSS `w-full h-full` |

## 💡 Naming Philosophy

> **Why "Obsidian Nirvana Handbook"?**
>
> - **Obsidian**: Your raw knowledge material, like unrefined obsidian ore.
> - **Nirvana**: In Buddhism, "parinirvana" means dying and being reborn — notes undergo three stages of fiery transformation, from static Markdown dead matter into breathing, interactive, co-living organisms within LiveView.
> - **Handbook**: Not a one-off script, but a reference guide you can open anytime, invoke on demand, and iterate continuously.
>
> Four fires:
> 1. **Obsidian → Marp**: From "writing for yourself" to "presenting to others"
> 2. **Marp → HTML**: From "local file" to "distributable format"
> 3. **HTML → LiveView**: From "one-way broadcast" to "real-time interaction"
> 4. **LiveView → Interactive Background**: From "silent black-and-white film" to "full audiovisual experience"
